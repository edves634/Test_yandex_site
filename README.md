## 1. Цель и объём тестирования

**Цель** – проверить сайт на соответствие макету Figma, качество вёрстки, производительность, адаптивность, кросс-браузерную совместимость и техническую корректность.

**Объём** – протестированы:
- Производительность (Lighthouse для десктопа и мобильных)
- Адаптивность (разрешения 320px, 375px, 768px, 1366px, 1920px)
- Кросс-браузерность (Chrome, Firefox, Яндекс.Браузер, Samsung Internet, Safari WebKit)
- Соответствие макету Figma (Pixel perfect на десктопе 1366px, мобилке 375px, частично 1920px)
- Качество кода (HTML/CSS/JS – семантика, брейкпоинты, логика скриптов)

### 4.2. Адаптивность на ключевых разрешениях

**📱 320px (iPhone SE / Firefox)**
- Заголовок не соответствует макету (размер, перенос строк)
- В блоке «Чтобы поддержать Международный васюкинский турнир» текст обрезается
- Карточка участника турнира обрезается по ширине
- **Вывод:** вёрстка для 320px отсутствует

**📱 375px (мобильный макет)**
- Шапка: высота 97px вместо ~70px – лишние 30px сдвигают контент
- Бегущие строки содержат только 1–2 фразы (должно быть 3)
- Футер: недостаточная высота, текст прижат к низу
- Декоративные слои (шахматы, эллипсы, марка) отсутствуют – вместо них одиночные фоновые изображения

**📱 768px (планшет)**
- Блок «Этапы преображения Васюков»: сетка отображается некорректно, текст в карточках обрезается справа
- Планшетный диапазон не проработан ни в макете Figma, ни в вёрстке – высокий риск ошибок

**💻 1366px (десктоп – базовая ширина)**
- Основные несоответствия: высота шапки (97px вместо 64px)
- Бегущие строки вложены в hero (должны быть после)
- Секция поддержки турнира содержит дублирующий блок .mob
- Отсутствуют фоновые слои
- Титул Остапа Бендера – «Чемпион мира» вместо «Гроссмейстер»

**💻 1920px (широкий экран)**
- Контент центрирован внутри max-width: 1366px – боковые поля не соответствуют макету Figma
- Логотип имеет смещение ~44px относительно макета
- Декоративные слои (жёлтый эллипс, шахматы, город) полностью отсутствуют

📸 **Скриншоты проблем** находятся в папке `screenshots` (файлы: `320px_issues.png`, `768px_timeline_broken.png`, `1920px_header_only.png`, `safari_webkit_375px_timeline.png`).

## 6. Качество кода (HTML/CSS/JS)

### 6.1. HTML – семантика и доступность

| Проблема | Наличие | Рекомендация |
|----------|---------|---------------|
| lang="en" при русском контенте | ❌ | Заменить на lang="ru" |
| Отсутствует `<main>` | ⚠️ | Обернуть основной контент |
| Ссылки-кнопки вместо `<button>` | ⚠️ | Для действий использовать `<button>` |
| Пустые href="#" у .btn__learn-more | ⚠️ | Удалить или заменить на реальные ссылки |
| Контрастность текста (например, .total-count) | ❌ | Увеличить контраст до ≥4.5:1 |
| Кнопки карусели без aria-label | ❌ | Добавить доступные имена |

### 6.2. CSS – брейкпоинты, адаптивность

| Проблема | Наличие | Рекомендация |
|----------|---------|---------------|
| Фиксация .container на max-width: 37.5rem | ❌ | Заменить на width: 100% + паддинги |
| Десктопный брейкпоинт с 768px (должен с 1024px) | ⚠️ | Сменить на min-width: 1024px |
| Планшетный диапазон (768–1024px) не проработан | ❌ | Добавить стили или медиа-запросы |
| Отсутствует адаптация под 320px | ❌ | Добавить @media (max-width: 374px) |
| Картинка .slider__card_img перекрывает текст (z-index) | ❌ | Уменьшить z-index / изменить позиционирование |

### 6.3. JavaScript – критические ошибки

| Проблема | Влияние | Приоритет |
|----------|---------|-----------|
| media.js перестраивает DOM при ресайзе (innerHTML) | Потеря обработчиков событий, конфликт слайдеров | 🔴 Критичный |
| slider.js и media.js конфликтуют: на десктопе уничтожается слайдер | Кнопки перестают работать | 🔴 Критичный |
| Авто-пролистывание карусели не отключается на мобильных | Невозможно остановить авто-скролл | 🟡 Средний |
| Сброс currentIndex=0 при ресайзе | Пользователь теряет позицию | 🟡 Средний |

**Рекомендация:** удалить `media.js`, всю адаптацию перенести на CSS Grid/Flex, отказаться от перестроения DOM на JavaScript.

## 8. Рекомендации по исправлению (по приоритетам)

### 🔴 Критичные / High (исправить в первую очередь)

1. **Устранить бесконечные запросы шрифта на 1920px** – проверить пути к Font Awesome, убрать динамическую генерацию версий
2. **Удалить media.js** – перенести всю адаптацию на CSS Grid/Flex
3. **Восстановить многослойные фоны** согласно Figma (абсолютное позиционирование, mix-blend-mode)
4. **Привести шапку к макету** – высота 64px на десктопе, логотип top:26px; на мобилке убрать фиксированную высоту
5. **Исправить бегущие строки** – добавить недостающие фразы (три фразы с разделителями-кружками), вынести из hero в отдельный блок
6. **Переделать секцию поддержки турнира** – оставить один блок с текстом и одним изображением, удалить лишний .mob
7. **Добавить адаптивность под 320px** – медиа-запрос, резиновые отступы, перенос текста

### 🟡 Средний приоритет

1. **Внедрить адаптивные изображения** – использовать srcset и sizes, конвертировать PNG в WebP/AVIF
2. **Настроить кэширование статики** (Cache-Control), добавить loading="lazy" для всех изображений
3. **Добавить правильные брейкпоинты** – 320px, 768px, 1024px, 1366px, 1920px
4. **Исправить карусель участников** – уникальные фото, круглый фон, верный титул Бендера, отключаемое авто-пролистывание
5. **Повысить контрастность текста** (проверить все серые элементы на белом фоне)
6. **Добавить семантические теги** (`<main>`, `aria-label`, `lang="ru"`)

### 🟢 Низкий приоритет / улучшения

1. Уменьшить количество JS-файлов, объединить их
2. Встроить критические CSS-стили, остальные подгружать асинхронно
3. Добавить предзагрузку шрифтов (`<link rel="preload">`)

## 9. Выводы

**Общая оценка качества:** ❌ Критически неудовлетворительно – сайт не готов к релизу.

**Два критических блокера к релизу:**
1. Бесконечные запросы шрифта на 1920px – сайт не загружается на широких экранах
2. Конфликт `slider.js` и `media.js` – карусель ломается при ресайзе, управление теряется

**Плюсы:**
- Базовый Lighthouse для десктопа показывает приемлемые значения (0.72)
- Вёрстка HTML/CSS в целом читаема, основные блоки присутствуют
- Карусель и слайдер визуально работают (игнорируя JS-конфликты)

**Минусы (критические):**
- Бесконечные запросы шрифта на 1920px
- Фундаментальное несоответствие макету Figma – потеряны ключевые визуальные элементы
- Полное отсутствие адаптивности под 320px и планшеты
- Серьёзные ошибки в JavaScript – media.js ломает работу при ресайзе
- Производительность на мобильных устройствах неудовлетворительная (LCP 7.8 с)

**Итог:** сайт требует полной ревизии вёрстки с упором на pixel perfect по макету Figma, переработки JS-логики (удаление media.js), исправления бесконечных запросов шрифта и добавления резиновой/адаптивной сетки. Только после этого можно проводить повторное тестирование и готовить к выпуску.

## 10. Вопросы для собеседования и ответы

### 10.1. Какие инструменты вы используете для кросс-браузерного тестирования и почему?

**Ответ:**  
Я использую комбинацию инструментов в зависимости от этапа тестирования:

1. **Локальные браузеры** – Chrome, Firefox, Яндекс.Браузер, Samsung Internet устанавливаются на тестовую машину (или VirtualBox). Это наиболее точный способ проверить базовую совместимость без задержек сети.
2. **BrowserStack / LambdaTest** – для проверки на реальных устройствах (iOS, Android, старые версии браузеров). Незаменимы для тестирования сенсорного управления, особенностей WebKit на разных версиях iOS и проблем с медиа-запросами на экзотических разрешениях.
3. **Playwright / Puppeteer** – для автоматизации скриншотных сравнений и регрессионного тестирования. Позволяют быстро проверить вёрстку на нескольких движках (Chromium, Firefox, WebKit) в CI/CD.
4. **Chrome DevTools (эмуляция устройств)** – для быстрой первичной проверки адаптивности, но не заменяет реальные устройства из-за различий в сенсорных событиях и производительности.

**Почему именно такой подход:**  
- Реальные браузеры (Firefox, Samsung Internet) часто имеют собственные баги рендеринга, которые не воспроизводятся в эмуляторах.  
- BrowserStack даёт доступ к сотням конфигураций без покупки физических устройств.  
- Playwright позволяет интегрировать кросс-браузерные проверки в автоматические тесты.

### 10.2. Как проверить адаптивность на реальных мобильных устройствах, если нет доступа к физическим телефонам?

**Ответ:**  
Доступ к физическим устройствам не всегда возможен, поэтому я использую комбинацию методов:

1. **BrowserStack / LambdaTest / Sauce Labs** – облачные сервисы, предоставляющие удалённый доступ к реальным iPhone, iPad, Samsung Galaxy и другим устройствам. Можно управлять пальцем (тач), поворачивать экран, эмулировать медленные сети и GPS.
2. **Эмуляторы Android Studio** – бесплатно, довольно точно воспроизводят поведение Chrome на Android, но не заменяют реальное устройство для теста аккумулятора и сенсора.
3. **Playwright с эмуляцией устройств** – позволяет задать user agent, размер экрана, touch-события, плотность пикселей (deviceScaleFactor). Подходит для автоматизации, но не выявит проблемы с производительностью на слабых чипах.
4. **Chrome DevTools (Device Toolbar)** – быстро проверить раскладку элементов на популярных разрешениях (iPhone SE, Pixel 5), но не покажет реальные touch-баги (например, 300 мс задержки).
5. **Remote debugging** – подключить реальный телефон по USB к Chrome/Firefox и инспектировать его через DevTools на компьютере. Так можно увидеть консоль, сеть, производительность, но тач-события всё равно придётся проверять пальцем.

**Мой подход:**  
Для экономии времени я сначала прогоняю адаптивность через Playwright и Chrome DevTools, нахожу 80% багов. Затем перед релизом прогоняю критический путь через BrowserStack на 5–6 реальных устройствах (iPhone 12, Pixel 4, Samsung Galaxy S21, iPad). Если совсем нет бюджета – прошу коллег или фокус-группу потестировать на своих телефонах.

### 10.3. Какие типы багов характерны для верстки на мобильных устройствах? Как их выявлять?

**Ответ:**  
**Характерные баги:**

1. **Обрезка текста/контента** – элемент не вписывается в ширину экрана, появляется горизонтальный скролл.  
   *Выявление:* уменьшать ширину окна, использовать эмуляцию iPhone SE (375px) и Galaxy Fold (280px).
2. **Неправильный размер тач-целей** – кнопки меньше 44×44px, сложно нажать пальцем.  
   *Выявление:* в Chrome DevTools включить «Tap highlighting» и проверить инспектором computed styles → `width`/`height`.
3. **Проблемы с зумом при фокусе на input** – при клике на поле ввода страница масштабируется, ломая вёрстку.  
   *Выявление:* проверить, что `font-size` на полях ввода ≥16px.
4. **Зависания/тормоза при скролле** – анимации или тяжёлые перерисовки.  
   *Выявление:* в Chrome DevTools → Performance → запись скролла, смотреть на FPS.
5. **Ложные срабатывания hover-эффектов** – на мобильных нет курсора, hover остаётся после касания.  
   *Выявление:* в эмуляторе устройств включить эмуляцию тача, проверить все интерактивные элементы.
6. **Проблемы с position: fixed** – на iOS fixed-элементы могут дёргаться при скролле.  
   *Выявление:* протестировать на реальном iOS через BrowserStack или удалённую отладку Safari.
7. **Фоновые видео/изображения не масштабируются** – выходят за границы или обрезаются.  
   *Выявление:* проверить `background-size: cover/contain` при разных соотношениях сторон экрана.

### 10.4. Какие подходы вы используете для анализа и улучшения производительности сайта?

**Ответ:**  

**Анализ:**
- **Lighthouse** (в Chrome DevTools и CI) – для метрик LCP, FID, CLS, TBT
- **WebPageTest** – детальный waterfall, сравнение с конкурентами
- **Chrome DevTools Performance** – запись и анализ длинных задач, перерисовок
- **Bundle analysis** – webpack-bundle-analyzer, source-map-explorer для JS
- **Network панель** – оценка размера ресурсов, времени блокировки, кэширования

**Улучшения (примеры):**
- **Оптимизация изображений** – конвертация в WebP/AVIF, добавление srcset и loading="lazy"
- **Критический CSS** – встройка стилей первого экрана, остальные загружать асинхронно
- **Сжатие текста** – Gzip/Brotli на сервере, минификация JS/CSS
- **Кэширование** – заголовки Cache-Control для статики, service worker для оффлайн-доступа
- **Уменьшение блокирующих ресурсов** – defer / async для скриптов, preload для шрифтов
- **Рефакторинг рендеринга** – избегать document.write, уменьшить количество перекомпоновок (reflow), использовать transform вместо position для анимаций

### 10.5. Как бы вы организовали процесс тестирования веб-приложений в вашей команде?

**Ответ:**  
Я бы построил процесс в соответствии с Agile/Scrum, интегрируя тестирование на всех этапах:

1. **Анализ требований** – участие QA в уточнении стори, проверка тестируемости, написание высокоуровневых тест-кейсов (TestRail / Qase)
2. **Разработка** – разработчик пишет юнит-тесты (Jest) и интеграционные тесты (Testing Library). QA пишет автотесты на API (Postman/Newman) и E2E (Playwright/Cypress)
3. **Проверка вёрстки и доступности** – используются axe-core, Lighthouse CI, скриншотные сравнения (Percy/BackstopJS)
4. **Регрессионное тестирование** – автотесты прогоняются на каждый PR (CI/CD – GitHub Actions / Jenkins). На ночь – полный набор smoke-тестов
5. **Ручное исследовательское тестирование** – перед релизом тестируем в браузерах / устройствах по матрице совместимости
6. **Отслеживание метрик** – сбор реальной производительности (RUM) через Sentry/New Relic, анализ ошибок и медленных запросов
7. **Обратная связь** – дефекты фиксируются в Jira с приоритетами. Проводятся ретроспективы для улучшения качества процессов

**Ключевые артефакты:** тест-план, чек-листы, баг-репорты, отчёты о прогонах автотестов, дашборд с метриками качества.
